# 5개 언어 비교: 같은 계산기, 다른 철학

같은 사양의 사칙연산 계산기를 Python → C → C++ → C# → Rust 순서로 구현하며 정리한 비교 문서입니다.

## 공통점

- 전체 구조가 동일: **루프 → 한 줄 입력 → 3조각으로 파싱 → 연산자 분기 → 출력**
- 모두 "잘못된 입력이 프로그램을 죽이면 안 된다"는 원칙으로 파싱 실패를 명시적으로 처리
- 연산자 분기는 전부 switch / match / elif 계열의 다중 분기 구문 사용
- 숫자는 전부 64비트 부동소수점(double / f64)

## 한눈에 보는 차이점

| 항목 | C | C++ | C# | Python | Rust |
|------|---|-----|----|--------|------|
| 실행 방식 | 컴파일(네이티브) | 컴파일(네이티브) | 컴파일(.NET JIT) | 인터프리터 | 컴파일(네이티브) |
| 메모리 관리 | 수동 | 수동(RAII 보조) | 가비지 컬렉터 | 가비지 컬렉터 | 소유권(컴파일 타임) |
| 타입 검사 | 정적(약함) | 정적 | 정적 | 동적 | 정적(강함) |
| 오류 처리 | 반환값 검사 | 스트림 상태 검사 | `TryParse` 패턴 | 예외(`try/except`) | `Result` + `match` |
| 코드 길이(체감) | 중간 | 중간 | 짧음 | **최단** | 최장 |
| 잘못된 입력 시 위험 | 전부 개발자 책임 | C보다 안전 | 런타임이 보호 | 런타임이 보호 | **컴파일러가 사전 차단** |

## 언어별: 무엇을 썼고, 왜 썼는가

### Python — `input` + `split` + `try/except`

"일단 시도하고 실패하면 예외 처리"하는 **EAFP** 스타일이 파이썬의 관용구.
`float()` 변환을 그냥 시도하고 `ValueError`를 잡는다. 코드가 5개 중 가장 짧고,
빌드 없이 수정 → 실행이 즉시 된다.

- 장점: 최단 코드, 최고의 개발 속도, 프로토타이핑 최강
- 단점: 타입 오류를 실행해야 발견, 실행 속도 최하, 배포에 Python 필요

### C — `fgets` + `sscanf`

`scanf`를 직접 쓰면 잘못된 입력이 stdin 버퍼에 남아 무한 루프가 생기기 쉬워서,
**한 줄을 통째로 읽고**(`fgets`) 그 문자열을 파싱(`sscanf`)하는 것이 안전한 표준 패턴.
반환값(`!= 3`) 검사가 유일한 오류 처리 수단이고, 빼먹어도 아무도 경고해주지 않는다.

- 장점: 가장 가볍고 빠름, 런타임 의존성 제로, 동작이 투명해 학습에 최적
- 단점: 버퍼 크기·반환값 검사 등 모든 안전장치가 개발자 몫, 실수하면 조용히 오동작

### C++ — `getline` + `istringstream`

C의 고정 버퍼 대신 `std::string`, `sscanf` 대신 문자열 스트림.
`iss >> a >> op >> b`의 성공 여부를 **bool 하나로** 확인할 수 있어 C보다 의도가 명확하다.
같은 문제를 C와 나란히 풀어보면 표준 라이브러리가 주는 안전성 차이가 확실히 보인다.

- 장점: C 수준 성능 + 메모리·파싱 안전성 개선, RAII, 거대한 생태계
- 단점: 언어가 방대해 같은 일을 하는 방법이 너무 많음, 에러 메시지 난해

### C# — `TryParse` + 최상위 문

예외를 던지는 `Parse` 대신 성공 여부를 bool로 돌려주는 **`TryParse`가 .NET의 표준 검증 패턴**
(Python의 EAFP와 정반대 접근). `string?` 널러블 타입 덕에 `ReadLine()`의 null 가능성도
컴파일러가 챙겨준다. 최상위 문으로 `Main` 보일러플레이트도 없앴다.

- 장점: 안전한 표준 패턴 내장, 짧은 코드 + 정적 타입, 도구 생태계(VS, NuGet) 우수
- 단점: .NET 런타임 필요, 단일 파일 배포 시 용량 큼, JIT 시작 지연

### Rust — `parse::<f64>()` + `match`

파싱 실패 가능성이 `Result` **타입 자체에** 들어 있어, 처리하지 않으면 값을 꺼낼 수 없다
= **오류 처리를 빼먹으면 컴파일이 안 되는 유일한 언어**.
`(Ok(a), op, Ok(b))` 패턴 매칭으로 두 숫자의 성공/실패를 한 줄에 분기한다.

- 장점: 컴파일러가 오류 처리 강제, GC 없이 C 수준 성능 + 메모리 안전성, cargo 도구 체인
- 단점: 코드가 가장 김, 소유권·빌림 학습 곡선 가파름, 컴파일 느린 편

## 결론

다섯 구현의 로직은 완전히 같다. 갈리는 지점은 단 하나 — **"오류를 어떻게 다루게 강제하는가"**:

- **C**: 개발자 재량 (검사를 안 해도 아무 일도 일어나지 않는다, 런타임에 터질 뿐)
- **C++**: 도구는 주지만 강제하지 않는다
- **Python / C#**: 런타임이 보호한다 (예외 / TryParse)
- **Rust**: 컴파일 타임에 강제한다 (처리 안 하면 빌드 실패)

용도별 선택: 빠른 프로토타이핑은 Python, 저수준 이해·임베디드는 C, 성능+추상화는 C++,
Windows·업무 생산성은 C#, 안전성과 성능을 다 원하면 Rust.
